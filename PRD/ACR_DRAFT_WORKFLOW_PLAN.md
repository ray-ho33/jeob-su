# PRD — 권익위 의결서 초안 작성 웹앱 워크플로우

> 작성일: 2026-06-11
> 범위: 구현 코드 변경 없이, 향후 웹앱 구현을 위한 PRD/구현계획만 정리합니다.

## 목적

`jeob-su`의 **Gemini 임베딩 기반 유사 의결례 검색**과 `korean-law-mcp`식 **공식 법령/결정문 조회·검증** 방식을 결합해, 국민권익위원회 의결서 초안을 작성하는 웹앱 워크플로우를 설계합니다.

이 웹앱은 사용자가 입력한 민원·사실관계·쟁점을 바탕으로 다음 일을 돕습니다.

1. 유사한 권익위 의결례를 찾습니다.
2. 관련 법령·공식 자료를 조회합니다.
3. 근거가 확인된 내용만 사용해 의결서 초안을 만듭니다.
4. 초안의 인용·근거·논리 흐름을 다시 검증합니다.

> **면책**: 이 도구는 법률 자문이나 최종 법적 판단을 제공하지 않습니다. 의결서 초안 작성과 근거 확인을 돕는 보조 도구이며, 최종 검토와 책임은 사람 담당자에게 있습니다.

## 먼저 알아야 할 용어

- **임베딩**: 긴 글을 의미가 담긴 숫자 목록으로 바꾸는 기술입니다. 비슷한 내용의 글은 숫자 목록도 비슷해집니다.
- **시맨틱 검색**: 단어가 똑같지 않아도 의미가 비슷한 문서를 찾는 검색입니다.
- **RAG**: 검색한 자료를 AI 답변에 함께 넣어, 근거 기반으로 답하게 하는 방식입니다.
- **MCP**: AI가 외부 도구를 안전하게 호출하도록 연결해 주는 규칙입니다.
- **Codex SDK**: 여러 AI 에이전트를 코드로 실행하고 조율하기 위한 기반입니다.
- **멀티에이전트**: 역할이 다른 여러 AI가 나누어 일하는 방식입니다.

## 전체 아키텍처

웹앱은 크게 5개 영역으로 나눕니다.

1. **프론트엔드**
   - 사용자가 민원 내용, 사실관계, 원하는 초안 형식을 입력합니다.
   - 검색 결과, 공식 조회 결과, 초안, 검증 결과를 화면에 보여줍니다.

2. **웹앱 백엔드**
   - 로그인, 프로젝트 저장, 초안 버전 관리, API 요청 처리를 담당합니다.
   - Codex SDK 멀티에이전트 실행을 시작하고 결과를 저장합니다.

3. **jeob-su 검색 도구**
   - 권익위 의결례 데이터를 준비합니다.
   - Gemini 임베딩으로 유사 의결례를 찾습니다.

4. **공식 조회/검증 도구**
   - `korean-law-mcp` 방식처럼 법제처 등 공식 출처에서 법령, 조문, 결정문 정보를 조회합니다.
   - 초안에 들어간 법령명, 조문, 사건번호, 인용 문구가 실제 자료와 맞는지 확인합니다.

5. **Codex SDK 멀티에이전트**
   - 검색, 법령 조회, 쟁점 정리, 초안 작성, 검증을 역할별로 나누어 수행합니다.

간단한 흐름은 다음과 같습니다.

```txt
사용자 입력
  -> 웹앱 백엔드
  -> Codex SDK 멀티에이전트 실행
  -> jeob-su 유사 의결례 검색
  -> 공식 법령/결정문 조회
  -> 근거 묶음 생성
  -> 의결서 초안 작성
  -> 검증 에이전트 재확인
  -> 웹앱 화면에 초안과 근거 표시
```

## 멀티에이전트 역할

| 에이전트 | 역할 | 산출물 |
|----------|------|--------|
| 접수 분석 에이전트 | 사용자의 민원 내용을 읽고 사실관계, 당사자, 요청사항, 핵심 쟁점을 나눕니다. | 쟁점 요약, 부족한 정보 질문 |
| 유사 의결례 검색 에이전트 | `jeob-su`의 Gemini 임베딩 검색으로 유사한 권익위 의결례를 찾습니다. | 유사 의결례 Top-K, 유사한 이유 |
| 공식 자료 조회 에이전트 | 공식 법령/결정문 조회 도구로 법령, 조문, 결정문 메타데이터를 확인합니다. | 공식 근거 목록 |
| 근거 정리 에이전트 | 검색 결과와 공식 조회 결과를 비교해 초안에 쓸 수 있는 근거만 추립니다. | 인용 후보, 근거 카드 |
| 초안 작성 에이전트 | 권익위 의결서 문체에 맞춰 초안을 작성합니다. | 의결서 초안 |
| 검증 에이전트 | 초안의 인용, 사건번호, 조문, 논리 비약, 미확인 주장을 확인합니다. | 수정 요청, 위험 표시 |
| 편집 에이전트 | 사람이 읽기 쉽게 문장, 항목, 표현을 다듬습니다. | 최종 검토용 초안 |

## 데이터 흐름

1. 사용자가 민원 본문과 기본 정보를 입력합니다.
2. 접수 분석 에이전트가 입력을 구조화합니다.
3. 유사 의결례 검색 에이전트가 구조화된 쟁점을 `search_similar_decisions`에 보냅니다.
4. 검색 결과의 결정문 ID를 기준으로 `get_decision_detail` 또는 `get_citation_pack`을 호출합니다.
5. 공식 자료 조회 에이전트가 법령명, 조문, 결정문 메타데이터를 공식 API로 다시 확인합니다.
6. 근거 정리 에이전트가 “검색으로 찾은 자료”와 “공식 조회로 확인된 자료”를 묶습니다.
7. 초안 작성 에이전트가 근거 묶음만 사용해 초안을 만듭니다.
8. 검증 에이전트가 초안과 근거 묶음을 대조합니다.
9. 웹앱은 초안, 근거, 검증 상태, 사람이 확인할 항목을 함께 보여줍니다.

개인정보와 민감정보 흐름도 함께 관리해야 합니다.

- 사용자가 입력한 민원 본문은 Gemini 임베딩 요청에 포함될 수 있습니다.
- 법령/결정문 조회 요청은 공식 API로 전송될 수 있습니다.
- 저장 전에는 주민등록번호, 전화번호, 주소 등 민감정보 마스킹을 권장합니다.

## RAG+공식조회 결합

이 워크플로우는 두 가지 방식을 함께 씁니다.

### 1. RAG로 후보를 넓게 찾기

`jeob-su`는 민원 텍스트와 의미가 비슷한 의결례를 찾는 데 강점이 있습니다.

- 장점: 표현이 달라도 비슷한 사안을 찾을 수 있습니다.
- 한계: “비슷하다”는 추천일 뿐, 공식 근거가 자동으로 보장되지는 않습니다.

### 2. 공식 조회로 근거를 좁히고 검증하기

`korean-law-mcp`식 공식 조회는 법령, 조문, 결정문 정보를 공식 출처에서 다시 확인하는 데 강점이 있습니다.

- 장점: 인용 가능성이 높은 공식 정보를 확인할 수 있습니다.
- 한계: 검색어가 정확하지 않으면 필요한 자료를 놓칠 수 있습니다.

### 결합 원칙

- RAG는 **후보 찾기**에 사용합니다.
- 공식 조회는 **근거 확인**에 사용합니다.
- 초안에는 **공식 조회 또는 원문 발췌로 확인된 내용**만 인용합니다.
- 유사도 점수만으로 결론을 내리지 않습니다.

## 의결서 작성 단계

의결서 초안은 다음 순서로 만듭니다.

1. **사건 정보 정리**
   - 신청인, 피신청인, 민원 요지, 관련 기관을 정리합니다.

2. **주문 후보 작성**
   - 의결서의 결론 부분입니다.
   - 예: 시정 권고, 의견 표명, 기각, 각하 등 후보를 제시합니다.

3. **이유 구조 만들기**
   - 사실관계
   - 당사자 주장
   - 관련 법령
   - 판단
   - 결론 순서로 뼈대를 만듭니다.

4. **유사 의결례 반영**
   - 비슷한 과거 의결례의 민원표시, 주문, 결정요지를 참고합니다.
   - 그대로 베끼지 않고, 현재 사건과 같은 점/다른 점을 표시합니다.

5. **공식 법령·결정문 검증**
   - 법령명, 조문 번호, 사건번호, 결정일자를 확인합니다.
   - 확인되지 않은 근거는 초안에서 제외하거나 “확인 필요”로 표시합니다.

6. **초안 작성**
   - 사람이 검토하기 쉬운 문단 형태로 작성합니다.
   - 각 주요 문단 옆에 근거 출처를 연결합니다.

7. **검증 리포트 생성**
   - 인용 확인 완료
   - 추가 확인 필요
   - 개인정보 마스킹 필요
   - 법률가 검토 필요 항목을 나눠 보여줍니다.

## API 설계

초기 API는 단순하게 설계합니다.

| 메서드 | 경로 | 역할 |
|--------|------|------|
| `POST` | `/api/drafts` | 새 의결서 초안 작업을 만듭니다. |
| `GET` | `/api/drafts/:draftId` | 초안 작업 상태와 내용을 조회합니다. |
| `POST` | `/api/drafts/:draftId/analyze` | 민원 본문을 구조화합니다. |
| `POST` | `/api/drafts/:draftId/search-cases` | 유사 의결례를 검색합니다. |
| `POST` | `/api/drafts/:draftId/official-lookup` | 공식 법령/결정문 정보를 조회합니다. |
| `POST` | `/api/drafts/:draftId/generate` | 근거를 바탕으로 초안을 생성합니다. |
| `POST` | `/api/drafts/:draftId/validate` | 초안의 인용과 논리 위험을 검증합니다. |
| `PATCH` | `/api/drafts/:draftId` | 사람이 수정한 초안을 저장합니다. |
| `POST` | `/api/drafts/:draftId/export` | 검토용 파일로 내보냅니다. |

예시 요청 구조:

```json
{
  "complaintText": "민원 본문",
  "requestedOutcome": "원하는 조치",
  "knownFacts": ["확인된 사실 1", "확인된 사실 2"],
  "privacyMode": "mask_before_external_call"
}
```

예시 응답 구조:

```json
{
  "draftId": "draft_001",
  "status": "needs_human_review",
  "draft": {
    "order": "주문 초안",
    "reasoning": "이유 초안"
  },
  "citations": [
    {
      "type": "acr_decision",
      "title": "유사 의결례 제목",
      "verified": true
    }
  ],
  "warnings": ["조문 인용 1건은 추가 확인이 필요합니다."]
}
```

## 구현 마일스톤

### 마일스톤 1: 요구사항과 데이터 경계 확정

- 민원 입력 항목을 정합니다.
- 외부 API로 보내도 되는 정보와 마스킹해야 하는 정보를 나눕니다.
- 완료 조건: 입력 폼, 저장 데이터, 외부 전송 데이터 기준이 문서화됩니다.

### 마일스톤 2: jeob-su 검색 연결

- 기존 MCP 도구 `health_check`, `ensure_semantic_corpus`, `search_similar_decisions`, `get_citation_pack` 호출을 연결합니다.
- 완료 조건: 웹앱에서 민원 본문으로 유사 의결례 Top-K를 볼 수 있습니다.

### 마일스톤 3: 공식 조회 연결

- 법령명, 조문, 결정문 메타데이터를 공식 조회 도구로 확인합니다.
- 완료 조건: 검색 결과와 공식 조회 결과가 근거 카드로 묶입니다.

### 마일스톤 4: Codex SDK 멀티에이전트 워크플로우

- 접수 분석, 검색, 공식 조회, 초안 작성, 검증 에이전트를 순서대로 실행합니다.
- 완료 조건: 하나의 민원 입력에서 검증 리포트가 포함된 초안이 생성됩니다.

### 마일스톤 5: 웹앱 편집 경험

- 초안 편집, 근거 확인, 경고 표시, 버전 저장을 구현합니다.
- 완료 조건: 사용자가 AI 초안을 수정하고 저장할 수 있습니다.

### 마일스톤 6: 내보내기와 운영 점검

- Markdown/DOCX/PDF 내보내기 중 1개 이상을 구현합니다.
- 로그, 비용, 오류 추적을 정리합니다.
- 완료 조건: 실제 검토 업무에 쓸 수 있는 파일을 만들 수 있습니다.

## 검증/평가

### 기능 검증

- 유사 의결례 검색 결과가 비어 있을 때도 친절한 안내가 나오는지 확인합니다.
- 공식 조회 실패 시 초안에 미확인 근거가 섞이지 않는지 확인합니다.
- Gemini API 키, 법제처 API 키가 없을 때 안전하게 중단되는지 확인합니다.

### 품질 평가

- 유사 의결례가 실제 쟁점과 관련 있는지 사람이 평가합니다.
- 초안의 주문, 이유, 판단 구조가 권익위 의결서 형식에 맞는지 확인합니다.
- 인용된 결정문과 법령이 원문과 일치하는지 확인합니다.

### 안전성 평가

- 개인정보가 외부 API로 그대로 전송되지 않는지 확인합니다.
- AI가 확인되지 않은 법령이나 사건번호를 만들어내지 않는지 확인합니다.
- 모든 초안에 “사람 검토 필요” 상태가 명확히 표시되는지 확인합니다.

### 추천 테스트 명령

현재 `jeob-su` 기준으로는 다음 명령을 개발 점검에 사용합니다.

```bash
node scripts/smoke-acr-mcp.mjs
```

이 명령은 MCP 서버가 기본 도구를 정상적으로 제공하는지 확인합니다.

## 리스크/대응

| 리스크 | 설명 | 대응 |
|--------|------|------|
| AI 환각 | 없는 법령, 사건번호, 판단을 만들어낼 수 있습니다. | 공식 조회로 확인된 근거만 인용하고, 검증 에이전트를 둡니다. |
| 유사도 오판 | 의미상 비슷해 보여도 법적 쟁점은 다를 수 있습니다. | 유사한 점과 다른 점을 함께 표시합니다. |
| 개인정보 전송 | 민원 본문에 개인정보가 들어갈 수 있습니다. | 외부 API 호출 전 마스킹 옵션을 기본값으로 둡니다. |
| API 비용 증가 | 임베딩, 검색, 초안 작성 호출이 많아질 수 있습니다. | 캐시, 호출 제한, 미리보기 모드를 둡니다. |
| 공식 API 장애 | 법령/결정문 조회가 실패할 수 있습니다. | 실패 시 초안 생성을 막거나 “확인 필요”로 표시합니다. |
| 책임 오해 | 사용자가 법률 자문으로 오해할 수 있습니다. | 모든 화면과 내보내기 문서에 작성 보조 도구임을 표시합니다. |
| 오래된 색인 | 의결례 데이터가 최신이 아닐 수 있습니다. | 색인 생성일과 데이터 범위를 화면에 표시합니다. |

## 우선순위

### P0: 반드시 먼저

- 개인정보 마스킹과 외부 전송 안내
- `jeob-su` 유사 의결례 검색 연결
- 공식 조회 기반 인용 검증
- 초안에 “법률 자문 아님, 사람 검토 필요” 표시
- 미확인 근거를 초안 본문에 확정 표현으로 넣지 않는 규칙

### P1: 첫 사용 가능한 버전

- 민원 입력 화면
- 유사 의결례 결과 화면
- 공식 근거 카드
- 의결서 초안 생성
- 검증 리포트
- 초안 저장과 버전 관리

### P2: 업무 효율 개선

- DOCX/PDF 내보내기
- 초안 비교
- 근거별 하이라이트
- 자주 쓰는 문구 템플릿
- 팀 검토 코멘트

### P3: 고도화

- 사건 유형별 작성 전략
- 기관별 표현 스타일 설정
- 사용자 피드백 기반 검색 품질 개선
- 운영 대시보드와 비용 분석

## 최종 완료 기준

- 사용자는 민원 본문을 입력해 유사 의결례와 공식 근거를 확인할 수 있습니다.
- AI는 확인된 근거를 바탕으로 의결서 초안을 작성합니다.
- 검증 리포트는 미확인 인용, 개인정보 위험, 사람 검토 필요 항목을 표시합니다.
- 웹앱은 최종 판단을 대신하지 않고, 작성 보조 도구라는 점을 명확히 알립니다.
